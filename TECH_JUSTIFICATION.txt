2. ARCHITECTURE DECISIONS & JUSTIFICATIONS
================================================================================

2.1 DATABASE LAYER: MySQL/MariaDB
────────────────────────────────────────────────────────────────────────────

WHY MYSQL?
  ✓ ACID Compliance: MySQL with InnoDB engine provides ACID (Atomicity, 
    Consistency, Isolation, Durability) guarantees critical for financial 
    transactions. This ensures payroll integrity - either a transaction 
    completes fully or rolls back entirely, preventing partial payments.
  
  ✓ Transaction Support: The SQL stored procedure ProcessPayroll() uses 
    START TRANSACTION/COMMIT/ROLLBACK to guarantee atomic operations, 
    ensuring salary calculations and shift status updates happen together 
    or not at all.
  
  ✓ Educational Alignment: As a database-focused course project, MySQL 
    demonstrates core database concepts: relational modeling, normalization, 
    foreign keys, and procedural SQL.
  
  ✓ XAMPP Integration: MySQL is bundled with XAMPP, eliminating complex 
    installation and configuration for classroom deployment.
  
  ✓ Scalability: While lightweight for a student project, MySQL easily 
    handles growth to production-scale payroll systems with millions of 
    records.

SCHEMA DESIGN DECISIONS:
  • InnoDB Engine: Specifically chosen for transaction support (MyISAM lacks 
    transactions and could result in data corruption during payroll runs).
  
  • Foreign Keys with CASCADE: Enforces referential integrity; deleting an 
    employee cascades to shifts and deductions, maintaining consistency.
  
  • DECIMAL(10,2) for Currency: Fixed-point arithmetic prevents floating-point 
    rounding errors critical in financial calculations (e.g., $1,234.567 would 
    truncate or round unpredictably with FLOAT).
  
  • Modular Deductions Table: Allows unlimited deduction types (SSS, PhilHealth, 
    loans, etc.) without schema modification, supporting business flexibility.


2.2 BACKEND: Python with mysql-connector-python
────────────────────────────────────────────────────────────────────────────

WHY PYTHON?
  ✓ Rapid Development: Python's simple syntax accelerates payroll business 
    logic implementation, allowing focus on algorithm correctness rather than 
    syntax complexity.
  
  ✓ Database Abstraction: mysql-connector-python library provides clean, 
    Pythonic database interaction without raw SQL string concatenation, 
    reducing SQL injection vulnerabilities.
  
  ✓ Modular Architecture: PayrollEngine class design encapsulates database 
    logic, enabling easy extension (e.g., adding RFID listeners, API 
    integrations) without modifying core payroll logic.
  
  ✓ Cross-Platform: Python runs identically on Windows, macOS, and Linux, 
    supporting deployment flexibility (e.g., running on a server, scheduling 
    with cron/Task Scheduler).
  
  ✓ Automation-Ready: Python's subprocess and scheduling capabilities (APScheduler, 
    cron) support automated daily/weekly payroll runs without manual intervention.

LIBRARY CHOICE: mysql-connector-python
  • Official MySQL driver ensuring full feature compatibility
  • Supports prepared statements preventing SQL injection
  • Connection pooling for scalable multi-request handling
  • Error handling through mysql.connector.Error for robust error management


2.3 WEB INTERFACE: PHP
────────────────────────────────────────────────────────────────────────────

WHY PHP?
  ✓ Server-Side Rendering: PHP processes on the server, generating HTML 
    without requiring client-side JavaScript framework complexity.
  
  ✓ XAMPP Ecosystem: Bundled with XAMPP; no additional installation or 
    configuration overhead for classroom environments.
  
  ✓ Direct Database Access: PDO (PHP Data Objects) enables clean database 
    queries within the same codebase as the web interface, reducing architectural 
    complexity.
  
  ✓ Form Processing: Built-in $_POST and $_GET superglobals simplify form 
    handling for employee CRUD operations, shift logging, and deduction management.
  
  ✓ Session Management: Native PHP sessions maintain user state across page 
    requests without explicit token management (simplicity vs. modern API 
    architectures).

DESIGN PATTERN: CRUD Portal
  • db.php: Database abstraction layer (functions like addEmployee(), logShift())
  • index.php: Request handling and view rendering
  • Separation of concerns: Database logic separated from presentation layer
  • Maintainability: Easy to extend with new employee/shift/deduction features


2.4 FRONTEND: JavaScript & CSS
────────────────────────────────────────────────────────────────────────────

WHY JAVASCRIPT?
  ✓ Client-Side Validation: script.js validates inputs before server submission:

WHY CSS?
  ✓ Responsive Design: CSS3 media queries ensure the portal works on desktop,

  ✓ Professional Appearance: Modern styling improves adoption; HR staff prefer

2.5 DEPLOYMENT PLATFORM: XAMPP

WHY XAMPP?

3. KEY TECHNICAL DECISIONS
================================================================================

3.1 TRANSACTION INTEGRITY (Most Critical for Payroll)
────────────────────────────────────────────────────────────────────────────

CHALLENGE: Payroll processing involves multi-step operations:
  1. Sum employee hours from shifts
  2. Calculate gross pay
  3. Calculate deductions (SSS, tax, etc.)
  4. Mark shifts as processed
  5. Store payroll record

RISK: If step 4 fails after step 3, shifts remain unprocessed but money is 
      deducted, causing data inconsistency.

SOLUTION: SQL Stored Procedure with START TRANSACTION/COMMIT/ROLLBACK
  
  Code Example (schema.sql):
  ──────────────────────────
  CREATE PROCEDURE ProcessPayroll(
      IN p_employee_id INT,
      IN p_start_date DATE,
      IN p_end_date DATE
  )
  BEGIN
      START TRANSACTION;
      
      -- Calculate payroll
      -- Insert records
      -- Mark shifts as processed
      
      COMMIT; -- All succeed or all rollback
  END;
  
  Benefits:
  • Atomicity: Entire payroll run succeeds or fails as a single unit
  • No inconsistent state: Database never shows partial updates
  • Audit trail: transaction_hash in payroll_records provides integrity verification
  • Prevents fraud: Impossible to partially process and hide transactions


3.2 MODULAR ARCHITECTURE
────────────────────────────────────────────────────────────────────────────

WHY MODULARITY?
  ✓ Separation of Concerns: Python backend, PHP portal, and SQL logic are 
    independent; changes to one layer don't require modifications to others.
  
  ✓ Extensibility: New features added without rewriting existing code:
    - Add RFID clock-in listener? Integrate into payroll_engine.py
    - Add email notifications? Extend PHP portal without touching Python
    - Add new deduction type? Update schema, no code changes needed
  
  ✓ Testing: Each module testable independently:
    - Unit test PayrollEngine.calculate_payroll() in isolation
    - Unit test PHP functions (addEmployee, logShift) without full XAMPP
    - Verify SQL logic with sample data before integration


3.3 CURRENCY PRECISION
────────────────────────────────────────────────────────────────────────────

WHY DECIMAL(10,2) NOT FLOAT?

Example Problem with FLOAT:
  Python:
    >>> 0.1 + 0.2 == 0.3
    False
    >>> 0.1 + 0.2
    0.30000000000000004
  
  Result: Payroll calculations accumulating rounding errors, causing employees 
  to lose (or gain) cents over time.

DECIMAL Solution:
  • Fixed-point arithmetic: No floating-point representation errors
  • Precision guaranteed: $1,234.56 is always exactly $1,234.56
  • Industry standard: Financial systems universally use fixed-point

4. ALTERNATIVE TECHNOLOGIES & WHY THEY WEREN'T CHOSEN
================================================================================

4.1 PostgreSQL Instead of MySQL
────────────────────────────────────────────────────────────────────────────
NOT CHOSEN because:
  • PostgreSQL not bundled with XAMPP; requires separate installation
  • More advanced features (JSON, arrays) unnecessary for payroll scope
  • MySQL sufficient and aligns with XAMPP ecosystem

Could be chosen in production if:
  • Greater concurrency required (PostgreSQL optimizes multi-writer scenarios)
  • Advanced reporting with complex queries needed
  • Integration with enterprise systems requiring PostgreSQL


4.2 Node.js/Express Instead of PHP
────────────────────────────────────────────────────────────────────────────
NOT CHOSEN because:
  • Requires npm, package management, build tools (complexity for classroom)
  • Node.js not bundled with XAMPP
  • JavaScript on backend and frontend is redundant for this scope

Could be chosen in production if:
  • Real-time updates needed (WebSockets)
  • Significant API consumption (REST/GraphQL endpoints)
  • Microservices architecture planned


4.3 NoSQL (MongoDB) Instead of SQL
────────────────────────────────────────────────────────────────────────────
NOT CHOSEN because:
  • Transaction support in MongoDB limited compared to SQL (until recent versions)
  • Financial data inherently relational: employees → shifts → deductions → payroll
  • Normalization prevents data redundancy
  • ACID guarantees critical for payroll integrity
  • Course emphasizes relational database design

Could be chosen in production if:
  • Highly unstructured data (varied employee records, custom fields)
  • Horizontal scaling across multiple servers required
  • Document-centric workflows (handling variable deduction types dynamically)


4.4 Django/FastAPI Instead of Python Script
────────────────────────────────────────────────────────────────────────────
NOT CHOSEN because:
  • Overkill for backend payroll engine; doesn't require web interface
  • Adds deployment complexity (WSGI server, Docker, etc.)
  • Payroll engine runs as scheduled task, not web service

Could be chosen in production if:
  • RESTful API required for third-party integrations
  • Multiple client interfaces (mobile app, desktop app) need backend API
  • Microservices architecture with multiple backend services

5. DEPLOYMENT & SCALABILITY PATH
================================================================================

CURRENT STATE (Educational):
  Architecture: Monolithic (all in XAMPP)
  Usage: Class project, ~50 employees max
  Reliability: Development-grade
  Security: Basic (no authentication)


PRODUCTION SCALABILITY PATH:

Phase 1 (Small Company, <500 employees):
  • Upgrade: Add SSL/TLS for HTTPS
  • Upgrade: Implement user authentication (bcrypt passwords)
  • Upgrade: Add logging/audit trails
  • Platform: Same XAMPP stack on dedicated server


Phase 2 (Medium Company, 500-5000 employees):
  • Split Architecture: Separate PHP portal server from Python payroll engine
  • Database: Dedicated MySQL server with replication/backup
  • Messaging: Add job queue (Celery/RabbitMQ) for asynchronous payroll runs
  • API: Expose Python backend as REST API for third-party integrations
  • Monitoring: Add application performance monitoring (APM)


Phase 3 (Large Enterprise, 5000+ employees):
  • Microservices: Separate services for employee management, shift tracking, 
    deduction processing, reporting
  • Cloud Platform: Deploy on AWS/Azure for geographic redundancy
  • Data Lake: Archive historical payroll data for analytics
  • Real-time: WebSockets for live dashboards
  • Security: OAuth2/OIDC authentication, encryption at rest

6. RISK MITIGATION
================================================================================

FINANCIAL DATA INTEGRITY:
  Risk: Payroll calculations errors → Employee disputes, legal liability
  Mitigation:
    • ACID transactions prevent partial updates
    • Database constraints (NOT NULL, CHECK) enforce valid data
    • Audit table logs all payroll changes with timestamps
    • transaction_hash enables verification of record integrity


SYSTEM AVAILABILITY:
  Risk: System down during pay period → Payroll delayed
  Mitigation:
    • Scheduled payroll (not real-time) allows offline processing
    • Database backups before each payroll run
    • Python script can be run from command-line if web portal unavailable


SECURITY:
  Risk: Unauthorized access to salary data → Privacy breach
  Mitigation (Future):
    • Add user authentication (login credentials)
    • Implement role-based access control (HR staff, managers, admins)
    • Encrypt sensitive data in transit (HTTPS) and at rest
    • Input validation (already in JS, needs server-side validation)


DATA ACCURACY:
  Risk: User input errors (negative hours, wrong dates)
  Mitigation:
    • JavaScript client-side validation
    • SQL constraints (hourly_rate > 0, clock_times in valid range)
    • Approval workflow for payroll runs before commit
    • Reconciliation reports comparing shifts to payroll

7. CONCLUSION
================================================================================

The technology stack (MySQL + Python + PHP + XAMPP) was selected to balance:

  1. EDUCATIONAL VALUE: Teaches relational databases, transactions, web 
     development, and backend scripting in a cohesive project.
  
  2. SIMPLICITY: XAMPP eliminates infrastructure complexity; students focus 
     on payroll logic, not system administration.
  
  3. FINANCIAL INTEGRITY: ACID transactions and fixed-point arithmetic ensure 
     payroll accuracy and reliability.
  
  4. EXTENSIBILITY: Modular architecture supports adding features without 
     rewriting components.
  
  5. REAL-WORLD RELEVANCE: Stack mirrors entry-level production payroll systems; 
     concepts transfer to enterprise systems.

Each technology choice was deliberate and justified by project requirements, 
educational objectives, and practical constraints. The architecture is 
scalable—paths exist to upgrade individual components (PHP → Node.js, MySQL 
→ PostgreSQL, XAMPP → Cloud) without rebuilding the entire system.
